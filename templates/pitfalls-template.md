# Project Pitfalls

> Long-term project pitfalls confirmed by the user or owner.
> Do not promote feature-local `lessons.md` items here without explicit
> confirmation.
> L0.5 knowledge files under `ai/knowledge/*` may reference confirmed pitfalls,
> but this file is the durable pitfall registry.

## Entry Format

- **ID**: PIT-YYYYMMDD-NNN
- **Area**: [module / workflow / device / UI / bridge / tooling]
- **Pitfall**: [what tends to go wrong]
- **Signal**: [how to notice it early]
- **Rule**: [what future specs/tasks/implementation must do]
- **Evidence**: [feature, commit, review, or owner confirmation]

## Promotion Rules

- A feature-local lesson must first appear in `workflow-record.md`,
  `improvement-candidates.md`, `promotion-report.md`, or `lessons.md`.
- Promotion requires explicit human approval.
- Each promoted pitfall must include source evidence, reason, and validation or
  owner-confirmed evidence.
- If evidence is missing, keep the lesson feature-local instead of adding it
  here.
